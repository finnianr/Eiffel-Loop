There are no assignment attempts in the main library code at all. All 17 instances are in the override/ directory, which contains patched copies of ISE's Vision2 and WEL libraries needed to work around bugs in ES 16.05. Each ?= converts to a standard if attached {TYPE} x as l_x then block — or may disappear entirely if the patches are no longer needed against a modern ISE Vision2. 

-- Before (non-void-safe)
a_pixmap_imp ?= Result.implementation

-- After (void-safe)
if attached {EV_PIXMAP_IMP} Result.implementation as a_pixmap_imp then
    ...
end

Two sub-categories

In code logic — lazy initialization and optional feature dispatch:

-- Lazy initialization (base/kernel/reflection)
if representation = Void then
    set_directly (object, string)
elseif attached {EL_STRING_FIELD_REPRESENTATION [...]} representation as r then
    set (object, r.to_value (string))
end

These require the attribute to be declared detachable (many already are, e.g. representation: detachable EL_FIELD_REPRESENTATION [...] in el_reflected_field.e:53). The Void guard then becomes: 

if not attached representation then
    set_directly (object, string)
elseif attached {EL_STRING_FIELD_REPRESENTATION [...]} representation as r then
    set (object, r.to_value (string))
end

In contracts — preconditions and postconditions:

-- Before
id_not_void: id /= Void
result_not_void: Result /= Void

-- After (attachment precondition / postcondition)
id_attached: attached id
result_attached: attached Result

All 7 instances of Result := Void are in routines whose return type is intentionally detachable. In most cases the detachable annotation is already present. The only change needed is confirming the return type declaration is explicit — the body Result := Void is then a no-op (default value) and can optionally be removed.

Pattern 4 — Lazy Initialization

Attributes that are optionally created on first use, currently detected as if x = Void then create x end.

Count in code logicFiles affected
22 occurrences28 files

There are two sub-patterns with different fixes:

a) Optional feature dispatch (attribute may never be set)

-- el_audio_clip_saver.e
sound_level_listener: EL_SIGNAL_LEVEL_LISTENER   -- not initialised in make

if sound_level_listener /= Void then
    sound_level_listener.set_signal_level (audio_rms_energy)
end

Fix: declare the attribute detachable and convert the guard.

sound_level_listener: detachable EL_SIGNAL_LEVEL_LISTENER

if attached sound_level_listener as sl then
    sl.set_signal_level (audio_rms_energy)
end

b) True lazy creation (attribute created on first access)

-- el_enumeration.e
if name_translater = Void then
    create name_translater.make (...)
end

In a newer EiffelStudio (where the ES 16.05 bug is fixed), this becomes a once ("OBJECT") function — cleaner and more efficient. See Pattern 6 below.

The ECF schema version also needs updating from configuration-1-16-0 to match the target EiffelStudio installation. Two ECF files already use the older configuration-1-15-0 schema (the GOBO base and one other) and will need updating regardless.

The is_attached_by_default = "false" attribute already appears on the override/ ECFs and the GOBO base ECF — exactly where legacy code requires it. This shows the ECF structure is already prepared for the transition.

Pattern 6 — EL_LAZY_ATTRIBUTE Workaround

There are 21 classes that inherit from EL_LAZY_ATTRIBUTE. This deferred class is a workaround for a specific bug in ES 16.05 where once ("OBJECT") did not work correctly. The class note documents this explicitly: 

-- This class is a workaround for bug in ES version 16.05.9
-- for the "once per object" construct:
--
--   object: MY_CLASS
--       once ("OBJECT")
--           create Result
--       end

EL_LAZY_ATTRIBUTE reimplements the same semantic using a detachable cached_item attribute and a manual null-check in lazy_item. This is already fully void-safe by design.

With a modern EiffelStudio, the bug is fixed. All 21 inheriting classes could replace: 

-- Before: using EL_LAZY_ATTRIBUTE workaround
inherit EL_LAZY_ATTRIBUTE

    new_item: MY_CLASS deferred end
    cached_item: detachable like new_item

-- Accessed via:
    result := lazy_item

with the native construct:

-- After: native once-per-object
my_object: MY_CLASS
    once ("OBJECT")
        create Result.make (...)
    end

This is a gain, not a cost: the workaround class and its boilerplate disappear entirely.

Recommended Migration Sequence

  1. Upgrade EiffelStudio before touching source code. The EL_LAZY_ATTRIBUTE bug is fixed in the compiler; the void-safe ISE base library is available; and you gain a void-safe Vision2 to compare against the override patches.
  2. Update all ECF schemas to match the new EiffelStudio version (global text replacement in the ECF files).
  3. Set void_safety = "transitional" in all ECF files. This mode flags issues as warnings rather than errors, letting the code compile while you work through the list.
  4. Add EL_HASH_TABLE.found_item redeclaration (one edit, resolves 165 call sites).
  5. Fix EL_HASH_SET_BASE probe sentinel (remove the redundant or old_key = Void branch).
  6. Work through the ~51 remaining files: add detachable to optional attributes and convert Void guards to attached checks.
  7. Assess the override/ directory: compare patches against the current ISE Vision2 source to decide whether to drop or convert.
  8. Optionally simplify EL_LAZY_ATTRIBUTE inheritors to use native once ("OBJECT").
  9. Switch to void_safety = "all" once transitional mode produces a clean compile.
