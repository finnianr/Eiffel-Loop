Void-Safety Migration: Pros and Cons

Analysis of the Eiffel-Loop library against EiffelStudio 16.05 (configuration-1-16-0, void_safety = "none" in all 57 ECF files). Covers 3,146 library source files plus 263 test files and 308 example files. All figures exclude the override/ directory unless stated.

Short answer: Very little capability is lost. The codebase already follows void-safe discipline in its core — the compiler switch has simply never been flipped. The remaining non-void-safe patterns are mostly deferred cleanup, with one genuine algorithmic issue in a custom hash-set probe loop.

1. The found_item Pattern — Biggest Usability Cost

There are 165 unguarded found_item usages following has_key throughout the codebase. Example from el_enumeration.e:131: 

if field_name_table.has_key (a_value) then
    Result := field_name_table.found_item   -- detachable G in void-safe mode
end

In void-safe mode, HASH_TABLE.found_item returns detachable G. The compiler cannot infer that has_key returning True guarantees attachment. Each call site would need an attached guard, which is verbose.

Recommended fix — one change, all 165 sites

Add a single covariant redeclaration to EL_HASH_TABLE (all 165 usages go through this class): 

found_item: G
    require
        item_found: found
    attribute end

The attached precondition found tells the compiler the item is guaranteed present, narrowing the type from detachable G to G. No call-site changes required.

2. EL_HASH_SET_BASE Probe Loop — the One Genuine Design Problem

In el_hash_set_base.e:288, the open-addressing probe routine uses Void as a sentinel for an empty slot:

old_key := area [index]
if old_key = default_key or old_key = Void then
    ...

With H -> HASHABLE, void-safe mode makes H attached by default. Comparing an attached type to Void is a compile error. More importantly, the class is already instantiated as EL_HASH_SET [CHARACTER_32] (an expanded type), where old_key = Void is a logical impossibility even today — void-safety merely surfaces this latent bug.

Fix

Remove the or old_key = Void branch. The existing deleted boolean array already tracks slot occupancy for all types. The = Void branch is redundant for reference types (covered by = default_key) and dead code for expanded types. This is self-contained in one class.

3. Optional Attribute Pattern — Minor Verbosity

About 22 "optionally set" attributes are declared without detachable but used with a Void guard. Example from el_audio_clip_saver.e: 

sound_level_listener: EL_SIGNAL_LEVEL_LISTENER    -- not marked detachable
    ...
if sound_level_listener /= Void then
    sound_level_listener.set_signal_level (audio_rms_energy)
end

Fix is purely mechanical: add detachable to the attribute declaration, and change the Void guard to if attached sound_level_listener as sl then. No capability is lost.

4. Type Anchor Patterns — Minor Awkwardness

Two slightly unusual patterns exist for expressing "optional callback" types:

OPTIONAL_ACTION in tp_factory.e

OPTIONAL_ACTION: detachable PROCEDURE [INTEGER, INTEGER]
    once
        Result := Void
    end

This is already void-safe — the return type is declared detachable and used as a type anchor for parameters like: 

unescaped_manifest_string (action_to_process_content: like OPTIONAL_ACTION): like all_of

No change needed here.

Result_consumer_type invariant in el_praat_audio_clip_analyzer.e:223

Result_consumer_purely_for_reference_as_an_anchor_type: Result_consumer_type = Void

This invariant documents that the type anchor feature is never actually called. With a detachable return type it remains trivially true but loses its documentary meaning as a guard. Replace with a comment. No capability lost.

5. Performance — No Meaningful Cost

There is no runtime overhead from void-safety for attached types. EiffelStudio proves attachment statically and generates no checks for proven-attached references. Runtime checks only occur for genuinely detachable types at the point of use — which is exactly where you would want them.

The ZSTRING implementation already uses the if attached area as l_area pattern heavily (15 occurrences in el_transformable_zstring.e alone). These guard against detachable SPECIAL areas inherited from ISE's READABLE_STRING_32. Once compiled against a void-safe ISE base library, EiffelStudio may eliminate several of these as provably redundant.

6. What You Would Gain

Eliminate the EL_LAZY_ATTRIBUTE workaround (21 classes)

The class EL_LAZY_ATTRIBUTE exists solely because of a bug in ES 16.05 where once ("OBJECT") did not work correctly. The class note says so explicitly: 

-- This class is a workaround for bug in ES version 16.05.9
-- for the "once per object" construct:
--
--   object: MY_CLASS
--       once ("OBJECT")
--           create Result
--       end

With a modern EiffelStudio, once ("OBJECT") works correctly. All 21 classes that inherit EL_LAZY_ATTRIBUTE could drop that inheritance and use the native construct directly. The cached_item attribute and the manual null-check in lazy_item disappear. The once ("OBJECT") mechanism is also more efficient — the runtime caches the result per object with no manual check on subsequent calls.

Contracts become more precise

Preconditions like x /= Void become attachment preconditions (attached x), which the compiler enforces structurally rather than treating them as assertions. This is a genuine semantic improvement.

The override directory may shrink or disappear

The override/ directory contains patched copies of ISE's Vision2 and WEL libraries (all 17 ?= assignment attempts are here). A modern EiffelStudio ships void-safe versions of these libraries. If the bugs that motivated the patches have been fixed upstream, the entire override directory could be dropped. If not, the 17 ?= calls are straightforward to convert to if attached {T} x as l_x then.

Recommended Migration Path

  1. Update EiffelStudio from 16.05 to a recent version first. The EL_LAZY_ATTRIBUTE bug is fixed in the compiler, and you get a void-safe ISE base library. Update ECF schemas from configuration-1-16-0 to the current version.
  2. Switch to void_safety = "transitional" across all 57 ECF files. This flags issues without requiring full compliance immediately and compiles most code without changes.
  3. Add the EL_HASH_TABLE.found_item redeclaration first — this resolves the largest single class of compiler warnings in one edit.
  4. Fix EL_HASH_SET_BASE probe sentinel detection.
  5. Mechanically address the remaining ~51 files: add detachable to optional attributes, convert = Void guards to attached checks.
  6. Assess the override directory: check whether the patches are still needed against the current ISE Vision2 library.
  7. Replace EL_LAZY_ATTRIBUTE inheritance in 21 classes with native once ("OBJECT").
  8. Switch to void_safety = "all" once transitional compiles cleanly.
